# v0.1.1 scope freeze

## 1. Purpose and authority

This document freezes the intended contents of the **v0.1.1** release. It exists so that "what is in v0.1.1" has one answer that everyone can point at, and so that adding late work is a deliberate, visible trade rather than quiet scope creep.

- **Linear is authoritative.** If an estimate or status here disagrees with the live Linear issue, the live issue wins. The values below are snapshots — see §7.
- **Mentioning an issue does not promote it.** Listing a backlog item for context (for example, to name a dependency or a cut candidate) does not commit it to the release. Only §3 is committed scope.

## 2. Dates and change control

- **Projected v0.1 ship:** 2026-08-09 (per `README.md` Status).
- **Scope review / grilling window:** 2026-08-10 through 2026-08-16.
- **Final freeze:** actual v0.1 ship date **+ 7 calendar days**. At the current projected ship date that is **2026-08-16**. If the ship date moves, the freeze date moves with it.
- **After the freeze**, adding an item requires one of:
  1. removing committed work of **equal or greater** estimate, or
  2. explicitly spending the reserved launch-week hot-fix capacity (§3).

## 6. Resolved implementation decisions (LLM bundle)

Two questions that used to be open are now locked. They are recorded here so the scope is not re-litigated during implementation.

- **Token estimation (WEG-140).** Ship `tiktoken-rs` for pre-call estimation behind a `$0.10`-per-cycle cap that aborts and falls back to deterministic mode on estimated overage. Anthropic tokenization differs from `tiktoken-rs` by roughly ±25%, so real cost may reach ~`$0.13` against the cap; the cap is designed to err toward false-positive aborts, and this inaccuracy is documented. No Anthropic count-tokens network call and no ported tokenizer at v0.1.1.
- **API-key storage (WEG-141).** The daemon reads API keys from `~/.config/dreamd/secrets.toml` at mode `0600`, never persisted into JSONL or autobiography commits. Service-install mode (launchd / systemd-user) reads that same path — no env-var dependency. OS keychain support is deferred (rejected for v0.1.1; revisit in v0.2).

**Reconciliation flag (WEG-137 vs WEG-141).** WEG-137 currently specifies reading API keys from environment variables.

That wording is superseded by WEG-141's later folded decision above — the `secrets.toml` path at mode `0600`, read identically under service-install mode. **WEG-137 must be reconciled to the WEG-141 decision before implementation begins.**
